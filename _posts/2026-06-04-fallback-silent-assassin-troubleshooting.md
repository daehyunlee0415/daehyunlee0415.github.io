고가용성(High Availability)을 위해 Fallback을 도입한 팀은 대부분 자랑스러워한다. 타임아웃이 나도 서비스는 멈추지 않고, 외부 API가 죽어도 파이프라인은 흐른다. 대시보드의 가용성 지표는 초록불을 유지한다. 이 글은 Fallback이 나쁘다고 말하려는 것이 아니다. 다만 **에러를 우아하게(gracefully) 처리하는 것이 항상 정답은 아니라는** 점을, 실무에서 수없이 반복해 본 경험을 바탕으로 짚고 싶다.

Fallback이 잘못 설계되면, 시스템은 '성공으로 위장한 실패(Success-disguised Failure)'를 낳는다. HTTP 200 OK가 반환되고, 파이프라인은 끊기지 않지만, 데이터 품질은 이미 훼손된 상태다. 장애의 시한폭탄은 그 순간이 아니라, 며칠 뒤 비즈니스 검증 단계에서 터진다. 그때 엔지니어는 로그를 뒤져도 원인을 찾기 어렵다. 왜냐하면 실패의 흔적이 이미 지워졌기 때문이다.

## Fallback의 양면성: 고가용성의 대가

Fallback의 근본적 목적은 분명하다. 단일 장애점(SPOF)을 제거하고, 무중단 서비스를 보장하며, 사용자 경험을 보호하는 것이다. 분산 시스템에서 이는 필수적인 방어 기제다. 누구도 "장애가 나면 그냥 죽여라"고 말하지 않는다.

문제는 **방어의 경계가 어디까지인가**다. 사용자에게 보이는 경험을 지키는 것과, 엔지니어가 시스템의 실제 상태를 파악하는 것은 다른 문제다. 전자만을 최적화하면 후자가 희생된다. 그리고 후자가 희생되면, 결국 전자도 무너진다. 다만 시차가 있을 뿐이다.

### 실무에서 마주치는 두 가지 시나리오

**LLM 평가 파이프라인.** 다단계 서브 질문(Sub-question)을 앙상블하여 최종 점수를 산출하는 복잡한 평가 프레임워크가 있다고 하자. 특정 모듈—예를 들어 일관성 검증기—가 타임아웃되면, 시스템은 조용히 기본값(Default Score: 0.5)으로 Fallback한다. 파이프라인은 멈추지 않고, 최종 평가 결과도 정상적으로 반환된다. 모델 팀의 대시보드에는 에러가 없다.

그러나 두 주 후, A/B 테스트에서 신규 모델의 품질이 유의미하게 저하되었다는 보고가 올라온다. 원인을 추적보면, 일관성 검증기의 타임아웃 빈도가 급증한 시점과 품질 저하 시점이 겹친다. 하지만 그 시점의 로그에는 에러가 없다. Fallback이 모든 것을 덮어버렸기 때문이다.

**시스템 트레이딩 파이프라인.** 실시간 시세 API가 지연되면, 파이프라인은 과거 캐시 데이터로 Fallback한다. 파이프라인은 멈추지 않고, 알파(Alpha) 모델은 계속 시그널을 생성한다. 인프라 팀의 모니터링 대시보드는 초록불이다. CPU, 메모리, 처리량—모두 정상이다.

그러나 PnL은 조용히 악화된다. 캐시된 시세로 계산된 시그널은 이미 시장과 어긋나 있다. 트레이딩 팀은 "모델이 망가졌다"고 결론 내리고, 모델 팀은 "인프라가 데이터를 제때 안 줬다"고 반박한다. 양쪽 모두 자기 영역의 지표는 정상이라고 주장할 근거가 있다. 이것이 Fallback이 만드는 **지표 환상(Metric Illusion)**이다.

## Fallback이 트러블슈팅을 지옥으로 만드는 기술적 이유

### 조용한 실패 (Silent Failures)

가장 치명적인 패턴은 겉으로는 성공이지만 실제로는 품질이 저하(Degraded)된 상태다. HTTP 200 OK, 정상적인 응답 페이로드, 에러 카운터의 무변화—모든 표면 지표가 정상을 가리킨다.

Root Cause 추적이 어려운 이유는 시간 지연 때문이다. 장애가 발생한 시점(T₀)과 비즈니스 영향이 드러나는 시점(T₀ + N일) 사이에 엄청난 간극이 생긴다. T₀ 시점의 로그는 이미 로테이션되어 사라졌거나, Fallback 처리로 인해 에러 레벨이 아닌 INFO 레벨로 기록되어 묻혀 있다. T₀ + N일 시점에 디버깅을 시작하면, 엔지니어는 결과만 보고 원인을 추론해야 한다. 이것은 인과 추론이 아니라 역사학이다.

### 연쇄적인 Fallback (Cascading Fallbacks)

Upstream 서비스가 Fallback 값을 반환하면, Downstream 서비스는 그 값이 정상인지 Fallback인지 구분할 수 없다. Downstream도 자체 Fallback 로직을 가지고 있다면, 이중, 삼중으로 Fallback이 중첩된다.

```
Service A (타임아웃) → default_value 반환
    ↓
Service B (default_value 수신, 품질 이상 감지 불가) → 자체 캐시 Fallback
    ↓
Service C (이중 Fallback 데이터로 계산) → 결과 반환 (HTTP 200)
```

이 시점에서 원인 추적의 맥락(Context)은 완전히 단절된다. Service C의 출력이 이상하다고 해도, Service A의 타임아웃이 근본 원인이라는 사실을 연결할 방법이 없다. 각 서비스의 로그는 독립적으로 "정상 처리"를 기록하고 있다.

### Fallback의 부패 (Fallback Decay)

메인 비즈니스 로직은 지속적으로 진화한다. 스키마가 바뀌고, 필드가 추가되고, 검증 규칙이 강화된다. 그러나 Fallback 로직은 좀처럼 호출되지 않기 때문에 방치된다.

6개월 전에 작성된 Fallback 핸들러는 당시의 스키마를 기준으로 작성되었다. 메인 로직은 3번의 리팩토링을 거쳤지만, Fallback 코드는 한 번도 실행되지 않아 리뷰 대상에서도 제외되었다. 그리고 어느 날, 메인 경로가 장애나 Fallback이 대량 발생하면—6개월 전의 시한폭탄이 터진다. Null Pointer Exception, 스키마 불일치, 잘못된 기본값으로 인한 연쇄 오류. **가장 오래된 코드가 가장 위험한 코드**가 되는 역설이다.

## 콘웨이의 법칙과 조직의 사일로

이 문제는 코드만의 문제가 아니다. 메릴린 콘웨이(Melvin Conway)의 법칙이 여기서도 작동한다: **시스템의 구조는 그것을 설계한 조직의 커뮤니케이션 구조를 반영한다.**

### Upstream과 Downstream의 단절

Upstream 팀이 에러를 조용히 덮어버리면, Downstream 팀은 데이터 품질 저하의 원인을 파악할 수 없다. Downstream은 "입력 데이터가 이상하다"고 보고하고, Upstream은 "우리 API는 200을 반환했다"고 반박한다. 양쪽 모두 자기 영역의 지표로는 무죄다.

이것이 **책임 전가(Blame Game)**의 구조적 원인이다. 기술적 문제가 조직적 갈등으로 승격된다. 그리고 갈등이 해결되기 전에, 비즈니스 영향은 계속 누적된다.

### 개발(Dev/Modeling)과 서빙(Serving/Ops)의 시야 차이

복잡한 알고리즘이나 모델을 개발하는 조직이 로직 내부에 자체 Fallback을 구현해 두는 경우가 흔하다. "모델 서빙 인프라가 불안정하니까, 우리가 알아서 처리하자"는 합리적인 판단이다. 모델 팀의 관점에서는 이것이 **자립성(Resilience)**이다.

그러나 인프라를 담당하는 서빙/Ops 조직의 대시보드에는 에러가 잡히지 않는다. 모니터링 사각지대가 생긴다. 인프라 팀은 "시스템이 정상이다"고 보고하고, 모델 팀은 "품질이 바닥이다"고 보고한다. 같은 시스템을 보고 있는데, 두 팀이 보는 현실이 다르다.

이 기이한 현상의 해결은 기술적 통합이 아니라 **관측성(Observability)의 공유**에서 시작된다. Fallback이 발생했을 때, 그 사실이 모델 팀의 로그뿐 아니라 인프라 팀의 메트릭에도 반드시 나타나야 한다. 그렇지 않으면 두 팀은 영원히 다른 세계를 살게 된다.

## 어떻게 구현할 것인가

### 방어적 실패 (Fail-Fast)의 도입

어설픈 기본값을 흘려보내는 것보다, 명시적으로 실패하게 놔두는 것이 더 안전한 구간이 있다. 특히 **데이터 무결성이 비즈니스 정확성에 직결되는 경계(Boundary)**에서는 Fail-Fast가 Fallback보다 낫다.

시스템 트레이딩에서 실시간 시세가 없으면 과거 캐시로 대체하는 것보다, 해당 시그널 생성을 건너뛰고 명시적으로 "데이터 부재" 상태를 기록하는 편이 낫다. PnL이 일시적으로 줄어드는 것과, 잘못된 시그널로 손실이 누적되는 것 중 어느 쪽이 더 나은 트레이드오프인가? 대부분의 경우 후자가 훨씬 비싸다.

Fail-Fast의 핵심은 "실패를 두려워하지 말고, 실패를 **보이게** 하라"는 것이다.

### 명시적 상태 전달 (Metadata Injection)

Fallback이 불가피한 경우—사용자 경험을 반드시 지켜야 하는 경우—응답에 대체 데이터임을 명시해야 한다.

```json
{
  "score": 0.5,
  "is_fallback": true,
  "degraded_reason": "consistency_checker_timeout",
  "fallback_timestamp": "2026-06-04T09:15:32Z",
  "original_error": "DeadlineExceeded: consistency_checker exceeded 30s timeout"
}
```

Downstream 서비스는 `is_fallback` 플래그를 확인하고, Fallback 데이터에 의존하는 계산의 신뢰도를 조정할 수 있다. 더 나아가, Fallback 비율이 임계치를 초과하면 Downstream이 자체적으로 알람을 발생시킬 수 있다. **맥락(Context)의 전파**가 연쇄 Fallback을 방지하는 유일한 방법이다.

HTTP 헤더를 활용하는 방법도 있다. `X-Degraded: true`, `X-Fallback-Reason: cache_stale` 같은 커스텀 헤더는 Downstream의 미들웨어에서 자동으로 캡처하고 메트릭으로 집계할 수 있다.

### 엄격한 예외 타겟팅

`catch(Exception)`과 같은 광범위한 예외 처리는 Fallback의 최악의 적이다. 모든 예외를 동일한 기본값으로 처리하면, 프로그래밍 버그와 일시적 네트워크 장애가 구분되지 않는다.

Fallback은 사전에 통제 가능한, 예측 가능한 실패에만 매핑해야 한다.

- 네트워크 타임아웃 (설정된 임계치 초과)
- 특정 HTTP 상태 코드 (503, 504)
- 서킷 브레이커 오픈 상태

이 외의 예외—NullPointerException, IllegalArgumentException, 스키마 파싱 오류—는 Fallback이 아니라 **즉시 실패**해야 한다. 이것들은 인프라 문제가 아니라 코드 버그다. 조용히 덮으면 안 된다.

## 어떻게 로깅/모니터링할 것인가

### 원본 에러의 보존

사용자/클라이언트 경험은 방어하되, 시스템 내부 로그에는 반드시 원인이 된 원본 에러 스택 트레이스(Original Stack Trace)를 남겨야 한다. Fallback이 발생했다는 사실만 기록하고 원본 에러를 버리는 것은, 환자에게는 "괜찮습니다"라고 말하면서 차트에는 아무것도 기록하지 않는 것과 같다.

```python
try:
    result = await consistency_checker.evaluate(input_data)
except TimeoutError as e:
    logger.warning(
        "Fallback triggered: consistency_checker timeout",
        extra={
            "fallback_type": "default_score",
            "fallback_value": 0.5,
            "original_error": str(e),
            "original_traceback": traceback.format_exc(),
            "correlation_id": correlation_id,
            "input_hash": hash_input(input_data),
        }
    )
    result = 0.5
```

상관관계 ID(Correlation ID)를 Fallback 로그에도 반드시 포함시켜, 요청의 전체 여정을 추적할 수 있게 해야 한다.

### 알람 피로 방지와 임계치 기반 알람

모든 Fallback을 `ERROR` 레벨로 찍으면 알람 피로(Alert Fatigue)가 온다. `INFO`로 찍으면 묻힌다. 이 딜레마의 해결책은 **Fallback을 별도의 커스텀 메트릭으로 분리**하는 것이다.

- `fallback_total{service="consistency_checker", reason="timeout"}` — 카운터
- `fallback_ratio` — 전체 요청 대비 Fallback 비율

알람 조건은 절대값이 아니라 비율 기반으로 설계한다. "Fallback이 1건 발생했다"가 아니라, "전체 트래픽의 5%를 초과하는 Fallback이 5분간 지속된다"일 때 알람이 울려야 한다. 그리고 그 알람은 **조직 간 공통 채널**—모델 팀과 인프라 팀이 모두 구독하는 채널—로 보내야 한다.

이것이 SLO(Service Level Objective) 관점에서의 Fallback 관측성이다. 가용성 SLO는 "서비스가 응답하는가"이고, 품질 SLO는 "서비스가 **올바르게** 응답하는가"다. 두 SLO를 분리하지 않으면, 가용성은 지키면서 품질은 잃는 시스템을 만들게 된다.

### 카오스 테스트 (Chaos Testing)

Fallback 로직이 최신 스키마를 유지하고 있는지 확인하는 유일한 방법은, 주기적으로 메인 경로를 의도적으로 실패시키는 것이다.

통합 테스트 파이프라인에서 분기별로 다음을 실행해야 한다:

- 특정 서비스에 인위적 타임아웃 주입
- 외부 API 모킹을 통한 503 응답 시뮬레이션
- Fallback 경로만 타도록 메인 경로 차단

Fallback이 발생했을 때 반환되는 데이터가 현재 스키마와 일치하는지, Null이 아닌지, Downstream이 소화할 수 있는 형태인지 검증한다. **의도된 실패 테스트(Intentional Failure Test)**가 Fallback Decay를 방지하는 유일한 예방책이다.

## 결론: 좋은 Fallback은 시끄러워야 한다

좋은 Fallback은 두 가지를 동시에 만족해야 한다.

**사용자에게는** 조용하게 에러를 덮어준다. 서비스는 중단되지 않고, 경험은 유지된다.

**시스템 관리자와 타 부서 엔지니어에게는** 이상 상태를 가장 명확하고 시끄럽게(Loud and Clear) 알려준다. 로그에 원본 에러가 보존되고, 메트릭에 Fallback 비율이 집계되고, 임계치 초과 시 조직 공통 채널에 알람이 울린다.

Fallback이 이 두 가지를 동시에 만족하지 못한다면, 그것은 고가용성이 아니라 **가용성의 환상**이다. 대시보드는 초록불이지만, 시스템은 이미 썩어가고 있다. 그리고 그 썩음은 조용해서, 냄새가 나기 전까지 아무도 모른다.

에러를 우아하게 처리하되, 시스템 안에서는 절대 우아하게 숨기지 마라. Fallback의 배신은 코드 한 줄에서 시작되지만, 그 대가는 조직 전체가 치른다.
